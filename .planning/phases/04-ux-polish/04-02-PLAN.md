Update AppShell.tsx to add two gates: an onboarding gate (outermost) and a mic explanation gate (within the game screen).

**1. Onboarding gate (AINP-04):**

Add import for OnboardingScreen. Add a selector for `hasSeenOnboarding`:

```typescript
import { OnboardingScreen } from './OnboardingScreen'

// Inside AppShell:
const hasSeenOnboarding = useBoundStore((s) => s.settings.hasSeenOnboarding)
```

Before the existing screen routing JSX, add a guard:

```tsx
if (!hasSeenOnboarding) {
  return (
    <div className="flex flex-col h-screen w-screen bg-gray-950 overflow-hidden">
      <OnboardingScreen />
      <KeyboardStrip />
    </div>
  )
}
```

This renders the onboarding screen with the keyboard strip still visible at the bottom. The guard runs BEFORE the existing screen switch statement, so no flash of level select occurs. This is the simplest hydration-safe approach (see research Pitfall 2).

IMPORTANT: Do NOT use `currentScreen` for this gate. Use `hasSeenOnboarding` directly — it is persisted, so it survives page refresh. `currentScreen` is NOT persisted and resets to `'levelSelect'` on every page load.

**2. Mic explain gate (AINP-05):**

Add import for MicExplainScreen. Add local state for permission check and explanation dismissal:

```typescript
import { MicExplainScreen } from './MicExplainScreen'

// Inside AppShell, after existing useState:
const [micPermission, setMicPermission] = useState<'unknown' | 'granted' | 'prompt' | 'denied'>('unknown')
const [micExplained, setMicExplained] = useState(false)
```

Add a useEffect to check mic permission state on mount:

```typescript
useEffect(() => {
  async function checkPermission() {
    try {
      const result = await navigator.permissions.query({ name: 'microphone' as PermissionName })
      setMicPermission(result.state as 'granted' | 'prompt' | 'denied')
    } catch {
      // Safari doesn't support permissions.query for microphone — default to 'prompt'
      setMicPermission('prompt')
    }
  }
  checkPermission()
}, [])
```

Modify the existing mic enable overlay inside the game screen section. Replace the current `{!micEnabled && (...)}` block with logic that checks whether to show the mic explain screen first:

```tsx
{/* Mic gate: explain screen -> enable button */}
{!micEnabled && (
  micPermission === 'prompt' && !micExplained ? (
    <MicExplainScreen onContinue={() => setMicExplained(true)} />
  ) : (
    <div className="absolute inset-0 flex items-center justify-center bg-black/60 z-30">
      <button
        onClick={() => setMicEnabled(true)}
        className="rounded-xl bg-white/10 border border-white/20 px-8 py-4 text-white text-lg font-semibold hover:bg-white/20 active:scale-95 transition-all backdrop-blur-sm"
      >
        Click to Enable Microphone
      </button>
    </div>
  )
)}
```

Flow:
- `micPermission === 'prompt'` AND `!micExplained` -> show MicExplainScreen
- After user clicks continue -> `micExplained = true` -> show the existing "Click to Enable Microphone" button
- `micPermission === 'granted'` -> skip explain, show enable button directly (user already granted in a previous session but page was refreshed)
- `micPermission === 'denied'` -> show enable button (will fail gracefully — existing error handling in useAudioInput covers this)
- `micPermission === 'unknown'` (still loading) -> show enable button (conservative default)

IMPORTANT: `micExplained` is local useState, NOT persisted. This is intentional — the explanation should show once per session when the permission is in 'prompt' state. If the user grants permission and reloads, the Permissions API returns 'granted' and the screen is skipped entirely.

IMPORTANT: Do NOT change anything about the ESC key handler, MIDI hook, or audio hook initialization. Only modify the mic enable overlay logic and add the onboarding gate.
