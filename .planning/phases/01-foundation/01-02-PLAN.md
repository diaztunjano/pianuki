<objective>
Wire the microphone pitch detection pipeline (Web Audio API + pitchy) and the MIDI input pipeline (Web MIDI API) as React hooks that dispatch unified NoteOn/NoteOff events to the Zustand audio store. Both hooks are activated in AppShell so the keyboard strip and debug panel immediately show live input.

Purpose: This plan delivers the core technical challenge of Phase 1 — real-time audio input from two sources producing identical event shapes. The hooks bridge imperative browser APIs (AudioContext, MIDIAccess) into the React state layer via Zustand's `.getState().addEvent()` pattern, keeping the hot audio path outside React's render cycle.

Output: A live audio pipeline where playing a note on an acoustic piano (via mic) or a MIDI keyboard causes the keyboard strip to light up and the debug panel to log the event — both sources producing identical InputEvent objects.
</objective>

  <action>
    Create `src/hooks/useMidiInput.ts` per research Pattern 5:
    - Call `navigator.requestMIDIAccess()` inside `useEffect`
    - For each input port, attach `onmidimessage` handler
    - Parse MIDI messages: `data[0]` status byte, `data[1]` note number, `data[2]` velocity
    - NoteOn: `(status & 0xF0) === 0x90 && velocity > 0`
    - NoteOff: `(status & 0xF0) === 0x80 || ((status & 0xF0) === 0x90 && velocity === 0)` — MUST handle velocity-0 NoteOn as NoteOff (research Pitfall 5)
    - For NoteOn: call `useBoundStore.getState().addEvent({ type: 'NoteOn', note, noteName: midiNoteToName(note), source: 'midi', timestamp: performance.now() })`
    - For NoteOff: call `useBoundStore.getState().addEvent({ type: 'NoteOff', note, noteName: midiNoteToName(note), source: 'midi', timestamp: performance.now() })`
    - Handle hot-plug: set `access.onstatechange` to re-attach handlers on all input ports when devices connect/disconnect
    - Cleanup: set all `input.onmidimessage = null` on unmount
    - Wrap `requestMIDIAccess()` in try/catch — if denied or unavailable, log warning but do not throw (MIDI is secondary input)

    Create `src/hooks/useAudioInput.ts` per research Pattern 4:
    - `useEffect` that calls an async `setup()` function
    - `setup()`:
      1. Call `navigator.mediaDevices.getUserMedia({ audio: true, video: false })` to get MediaStream
      2. Create `new AudioContext()` — MUST be inside the getUserMedia success path (research Pitfall 2: AudioContext requires user gesture)
      3. Create `AnalyserNode` with `fftSize = 2048`
      4. Connect `MediaStreamSource` → `AnalyserNode`
      5. Create `PitchDetector.forFloat32Array(analyser.fftSize)` from pitchy
      6. Allocate `Float32Array(analyser.fftSize)` buffer
    - Poll loop via `requestAnimationFrame`:
      1. Read time-domain data: `analyser.getFloatTimeDomainData(buffer)`
      2. RMS silence gate: compute RMS of buffer, if below `SILENCE_THRESHOLD` (0.01), emit NoteOff for lastNote if any, skip detection
      3. Call `detector.findPitch(buffer, context.sampleRate)` → `[frequency, clarity]`
      4. If `clarity > CLARITY_THRESHOLD` (0.9) AND frequency in piano range (27Hz–4200Hz):
         - Convert to MIDI note via `frequencyToMidiNote(frequency)`
         - If different from lastNote: emit NoteOff for previous lastNote (if any), then emit NoteOn for new note
      5. If clarity below threshold AND lastNote is not null: emit NoteOff for lastNote
      6. All events dispatched via `useBoundStore.getState().addEvent(...)` with `source: 'mic'`, including `frequency` and `confidence: clarity` fields
    - Define named constants at module level: `CLARITY_THRESHOLD = 0.9`, `FFT_SIZE = 2048`, `SILENCE_THRESHOLD = 0.01`
    - Cleanup function: `cancelAnimationFrame(frameId)`, stop all stream tracks, close AudioContext
    - Export the hook as a named function: `export function useAudioInput()`
    - The hook should accept an optional `enabled: boolean` parameter (default `true`) — when false, skip setup entirely. This lets the UI add a "Start Mic" button if needed to handle the user gesture requirement.
  </action>
